# PocketVibe：基于QLoRA微调的移动端微应用代码生成器部署与评估报告

---

## 一、引言（Introduction）

本报告介绍PocketVibe项目的个人部署与评估。PocketVibe是一个基于**Qwen2.5-Coder-1.5B-Instruct**大语言模型，通过QLoRA（Quantized Low-Rank Adaptation）微调技术训练的移动端微应用代码生成器。用户输入一句中文自然语言描述（如"做一个倒计时器"），模型即可直接生成一个完整的、可独立运行的HTML/CSS/JavaScript文件，适配手机屏幕显示。

该项目部署于香港教育大学HPC集群（aaillm.eduhk.hk），使用NVIDIA A16 GPU（16GB VRAM）进行训练与推理。本报告将依次介绍部署流程与系统配置（第二节）、模型在多场景下的测试评估（第三节）、基于评估结果的改进建议（第四节），以及对课程学习成果的反思（第五节）。

---

## 二、部署（Deployment）

### 2.1 系统规格

| 组件 | 规格 |
|------|------|
| GPU | NVIDIA A16, 16GB VRAM |
| CPU | 8 cores |
| 系统内存 | 32GB RAM |
| 操作系统 | Linux (HPC集群) |
| 软件栈 | PyTorch 2.9.1, CUDA 12.6, Transformers 4.57.1, PEFT, TRL |
| 存储配额 | 25GB (/home/studentXX/) |

### 2.2 部署步骤

整个部署流程分为数据准备、模型训练、推理服务三个阶段：

**阶段一：训练数据准备（本地）**

1. 运行`scripts/00_create_seeds.py`手工编写50条高质量种子数据（指令+完整HTML代码三元组）
2. 运行`scripts/01a_augment_instructions.py`通过DeepSeek-V3 API对每条种子生成10个指令变体（共500条）
3. 运行`scripts/01b_generate_new_tools.py`生成90条全新工具类型的完整代码配对
4. 运行`scripts/01c_merge_all.py`合并去重，得到约550条训练数据
5. 运行`scripts/02_validate_data.py`进行质量检验（HTML完整性、viewport标签、长度、无外部CDN）
6. 运行`scripts/02b_split_data.py`按9:1划分训练集与验证集

**阶段二：QLoRA微调训练（HPC）**

7. 通过SCP上传项目至HPC：`scp -r Enoch student@aaillm.eduhk.hk:~/PocketVibe`
8. 提交训练作业：`sbatch slurm/train.slurm`
9. 训练配置：4-bit NF4量化加载基座模型（~1.5GB显存），LoRA rank=32, alpha=64，目标模块为Attention层（q/k/v/o_proj），有效batch size=8（梯度累积），余弦学习率衰减，训练4个epoch，总耗时约53分钟

**阶段三：推理部署（HPC）**

10. 提交推理服务：`sbatch slurm/serve.slurm`，启动FastAPI服务（port 8000）
11. 通过Gradio前端（port 7861）提供用户界面，支持输入指令→生成代码→实时预览

> **【此处插入截图1】**：Gradio前端界面截图，展示输入框、生成按钮、HTML预览区域。
> 文件位置：如果已有截图，放在 `report/` 目录下；如果没有，需要登录HPC启动服务后截图。

### 2.3 模型选择论证

我对比了三个候选模型：

| 模型 | 量化后大小 | 中文支持 | 代码专精 | 选择理由 |
|------|-----------|---------|---------|---------|
| **Qwen2.5-Coder-1.5B-Instruct** ✅ | ~1.5GB | 原生支持 | ✅ 代码预训练 | 显存余量充足，可用大rank和长序列 |
| Qwen2.5-Coder-7B | ~4GB | 原生支持 | ✅ | A16上LoRA rank受限，训练不稳定 |
| CodeLlama-7B | ~4GB | 需适配 | ✅ | 不原生支持中文，需额外数据适配 |

选择1.5B模型的核心原因：4-bit量化后仅占1.5GB显存，在A16的16GB上留出充足空间给LoRA训练参数、KV Cache和2048长度序列，确保训练和推理均稳定运行。

## 三、评估（Evaluation）

### 3.1 评估方法

采用8维度自动评分体系（每维度0-2分，满分16分简化为10分制）对比基座模型（未微调）与微调模型在相同指令下的输出质量。评估维度包括：HTML完整性、无Markdown包裹、Viewport适配、风格匹配（渐变+圆角）、JS功能完整性、无外部CDN依赖、代码长度合理性、中文界面。

### 3.2 测试场景与结果

**场景1：简单工具——"做一个秒表，有开始、停止和清零按钮"**

| 指标 | 基座模型 | 微调模型 |
|------|---------|---------|
| 总分 | 4/10 | **9/10** |
| HTML完整 | ✗ | ✓ |
| 风格匹配 | 无渐变无圆角 | 渐变背景+圆角卡片 |
| JS功能 | 基本可用 | 完整（开始/停止/清零） |

> **【此处插入截图2】**：微调模型生成的秒表HTML在手机浏览器中的渲染效果。
> 来源文件：`data/eval/test_1.html`，用手机浏览器打开后截图。

**场景2：复杂工具——"做一个番茄钟，暗色主题，自动在工作和休息之间切换，记录今日完成数"**

| 指标 | 基座模型 | 微调模型 |
|------|---------|---------|
| 总分 | 5/10 | **9/10** |
| 风格匹配 | 部分暗色 | 完整暗色渐变主题 |
| JS功能 | 仅基础计时 | 自动切换+轮次记录 |

**场景3：模糊输入——"我要个能记事的小本本"**

| 指标 | 基座模型 | 微调模型 |
|------|---------|---------|
| 总分 | 4/10 | **7/10**（最低分） |
| JS功能 | 无保存逻辑 | 有localStorage保存 |
| 风格匹配 | 无 | 有渐变但较简单 |

### 3.3 优势总结

1. **格式合规性显著提升**：微调后10/10测试案例均通过HTML完整性、无MD包裹、无外部CDN检查（基座模型0/10通过HTML完整性）
2. **UI风格一致性**：所有输出均包含渐变背景、圆角卡片、适当阴影，符合训练数据中的现代移动端设计规范
3. **功能理解力**：对"暗色主题""自动切换""记录完成数"等修饰词能准确响应

### 3.4 劣势分析

1. **模糊输入泛化不足**："我要个能记事的小本本"仅得7分，模型缺乏对非具体指令的创意补全能力
2. **推理速度慢**：使用HuggingFace Transformers原生generate()，每次生成约30-90秒，用户体验欠佳
3. **Epoch 2后过拟合**：验证集loss在epoch 2达到最低（0.190），之后回升至0.225（epoch 4），说明550条数据量不足以支撑更多轮训练

## 四、改进建议（Recommendations）

### 建议1：扩充训练数据量（对应劣势3——过拟合）

当前仅550条训练数据导致epoch 2即出现过拟合。建议：
- 将数据扩充至3000-5000条，覆盖更多工具类型和指令表述风格
- 引入"模糊指令→高质量代码"配对数据，专门训练模型的创意补全能力
- 预期效果：可训练5-8个epoch而不过拟合，进一步降低eval_loss

### 建议2：迁移至vLLM推理框架（对应劣势2——速度慢）

当前使用HuggingFace原生generate()无法利用KV Cache优化和连续批处理。建议：
- 迁移至vLLM框架，利用PagedAttention和CUDA Graphs加速
- 预期效果：推理速度提升15-24倍，从30-90秒降至3-5秒
- 实施成本低：vLLM原生支持PEFT adapter加载

### 建议3：引入DPO偏好优化（对应劣势1——模糊输入泛化）

对于模糊输入场景，模型需要学会"补全"用户未说明的需求细节。建议：
- 构建偏好数据对：同一模糊输入下，人工标注"好输出"和"差输出"
- 使用DPO（Direct Preference Optimization）进行第二阶段训练
- 预期效果：模糊输入场景得分从7分提升至8-9分

## 五、反思（Reflection）

### 5.1 课程学习成果对照

本项目直接实践了课程的多个CILO：

- **CILO1（理解多模态LLM部署基础原理）**：通过实际配置4-bit NF4量化、理解量化级别在正态分布上的放置原理，深刻理解了QLoRA如何在有限硬件资源上实现大模型部署。
- **CILO2（分析LLM部署的系统需求）**：在模型选择阶段系统评估了1.5B vs 7B模型在A16 GPU上的显存占用、LoRA参数空间和序列长度的权衡关系。
- **CILO3（监控和维护AI服务器）**：通过`nvidia-smi`监控GPU显存使用、Slurm作业队列管理、训练日志分析等实践，掌握了AI服务器运维技能。

### 5.2 伦理考量

PocketVibe生成的代码在用户设备上本地运行，不涉及用户数据上传，隐私风险较低。然而，模型可能生成存在安全漏洞的JavaScript代码（如XSS），未来需引入代码安全审计模块。

### 5.3 未来应用场景

通过GGUF格式量化，该1.5B模型可部署至移动设备本地运行（参考Gemma在iPhone上的实践），实现完全离线、零延迟、隐私保护的个人微应用生成器——这与"Personal LLM"课程主题高度契合。
